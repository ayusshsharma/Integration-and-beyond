1. Executive Summary

This guide documents a hybrid encryption pattern for protecting sensitive API payloads in transit beyond TLS. Each request generates a fresh AES-256 session key and a random 16-byte IV. The JSON payload is encrypted with AES-256-CBC; the AES key is then encrypted with the receiver’s X.509 RSA public key.

The encrypted AES key travels in the HTTP header sessionKey, the IV in header iv, and the ciphertext in the request body. The backend recovers the AES key with its RSA private key, then decrypts the payload with AES key + IV before business processing.

Design intent

Combine RSA’s key-distribution strength with AES’s performance for large payloads—without ever sending a plaintext AES key over the wire.

4. Encryption Workflow

  1. Generate AES-256 session key — cryptographically secure random 32-byte key (per request).
  2. Generate IV — cryptographically secure random 16-byte initialization vector (per request).
  3. Encrypt payload — AES-256-CBC over the JSON body (e.g. {"data":"test"}).
  4. Wrap AES key — encrypt the AES key with the receiver’s RSA public key from an X.509 certificate.
  5. Assemble HTTP request — set sessionKey and iv headers; place ciphertext in the body.
  6. Transmit over HTTPS — TLS protects headers and body in transit.
  7. Backend unwrap & decrypt — RSA private key recovers AES key; AES + IV recovers plaintext.
  8. Process business request — validated plaintext is ready for application logic.
Non-negotiable rule

Generate a new AES key and a new IV for every encryption operation. Never reuse either value across requests.

5. Frontend Process

Client-side responsibilities (browser, mobile, or BFF acting as encrypting client):

  1. Load and trust the receiver’s X.509 RSA public certificate (pin or fetch from a controlled endpoint).
  2. Generate a random AES-256 key using a CSPRNG (crypto.getRandomValues / platform equivalent).
  3. Generate a random 16-byte IV.
  4. Serialize the business JSON, then encrypt with AES-256-CBC using the session key and IV.
  5. Base64-encode (or otherwise encode as agreed) the ciphertext and IV.
  6. RSA-encrypt the AES key with the public key; encode for the sessionKey header.
  7. Send POST (or required verb) with headers + encrypted body over HTTPS.
// Pseudocode — frontend encryption steps
const aesKey = generateAes256Key();          // 32 bytes, CSPRNG
const iv     = generateIv();                 // 16 bytes, CSPRNG
const cipher = aes256CbcEncrypt(json, aesKey, iv);
const wrappedKey = rsaEncrypt(aesKey, receiverPublicKey);

headers["sessionKey"] = base64(wrappedKey);
headers["iv"]         = base64(iv);
body = { encryptedPayload: base64(cipher) };

6. Backend Process

Server-side responsibilities:

  1. Accept HTTPS request; extract sessionKey, iv, and encrypted body.
  2. Decode header values from Base64 (or agreed encoding).
  3. Decrypt sessionKey with the RSA private key to recover the AES session key.
  4. Decrypt the body ciphertext with AES-256-CBC using recovered key + IV.
  5. Parse JSON, validate schema/business rules, process the request.
  6. Return a normal (or similarly protected) API response per contract.
Success state

When decryption and validation succeed, the payload is marked Ready for Processing and handed to business services.

7. IV Handling

The Initialization Vector (IV) randomizes ciphertext so identical plaintexts never produce identical ciphertexts under the same key. CBC mode requires a unique IV per encryption with a given key.

Figure 2 — IV lifecycle across encrypt → transmit → decrypt

  • IV length for AES-CBC: 16 bytes (128 bits).
  • IV is transmitted in clear (Base64) via the iv header—confidentiality of IV is not required.
  • Uniqueness (and unpredictability) of IV is required; reuse with the same key is a critical vulnerability.

8. Request Structure

Canonical request shape for encrypted business calls:

POST /api/customer HTTP/1.1
Host: api.example.com
Content-Type: application/json
sessionKey: <RSA Encrypted AES Key>
iv: <Base64 Encoded IV>

{
  "encryptedPayload": "<AES Encrypted Cipher Text>"
}
Element Location Content
sessionKey HTTP header AES-256 key encrypted with receiver RSA public key (typically Base64)
iv HTTP header 16-byte IV, Base64-encoded
encryptedPayload JSON body AES-256-CBC ciphertext of original JSON (Base64 or agreed encoding)

9. Decryption Process

  1. Read sessionKey and iv from headers; read encryptedPayload from body.
  2. Base64-decode all three values.
  3. RSA decrypt sessionKey with the private key → plaintext AES-256 key.
  4. AES-256-CBC decrypt ciphertext using AES key + IV → original JSON bytes.
  5. UTF-8 decode / JSON parse → e.g. {"data":"test"}.
  6. Authorize, validate, and process the business request.
// Pseudocode — backend decryption steps
const wrappedKey = base64Decode(headers.sessionKey);
const iv         = base64Decode(headers.iv);
const cipher     = base64Decode(body.encryptedPayload);

const aesKey    = rsaDecrypt(wrappedKey, privateKey);
const plaintext = aes256CbcDecrypt(cipher, aesKey, iv);
const payload   = JSON.parse(plaintext);
// → { "data": "test" }

10. Security Considerations

New AES key per request

Never reuse AES session keys across requests or messages.

New IV per request

Never reuse an IV with the same AES key. Generate a fresh 16-byte IV every time.

Protect RSA private key

Store private keys in HSM/Key Vault/KMS. Restrict process access and rotate on schedule.

Always use HTTPS

TLS is mandatory. Hybrid encryption complements—not replaces—transport security.

AES key via RSA public key

Only the holder of the matching private key can recover the session key.

Payload via AES-256-CBC

Bulk data stays on the fast symmetric path after the key is wrapped asymmetrically.

Additional hardening recommendations

Prefer authenticated encryption (e.g. AES-GCM) if the platform allows a future evolution; with CBC, ensure PKCS#7 padding validation is constant-time and consider an HMAC or AEAD layer to prevent padding-oracle class attacks. Pin certificates where feasible; reject weak RSA key sizes (< 2048 bits).

11. Implementation Checklist

  • ✓
    Certificate distribution — Frontend has the correct receiver X.509 RSA public key; trust store / pin defined.
  • ✓
    CSPRNG — AES key (32 bytes) and IV (16 bytes) generated with a cryptographically secure RNG.
  • ✓
    AES-256-CBC encrypt — Payload encrypted; ciphertext encoding agreed (Base64 recommended).
  • ✓
    RSA wrap — AES key encrypted with public key; placed in sessionKey header.
  • ✓
    IV header — Base64 IV sent in iv header; never omitted.
  • ✓
    HTTPS only — Cleartext HTTP rejected in all environments (including lower envs where possible).
  • ✓
    Private key custody — Backend private key in vault/HSM; no keys in source control or logs.
  • ✓
    Decrypt path — Backend unwraps AES key, decrypts payload, validates JSON schema.
  • ✓
    No reuse — Unit/integration tests assert new key + IV per request.
  • ✓
    Observability — Log correlation IDs only; never log AES keys, private keys, or raw sensitive plaintext in shared logs.
  • ✓
    Failure handling — Uniform error responses for decrypt failures to reduce oracle leakage.
  • ✓
    Documentation handoff — ADD / SDR / client SDK notes updated with header names and example payloads.
